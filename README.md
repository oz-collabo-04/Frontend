### Zustand - 상태 관리 라이브러리

- 이점: 간단한 API와 가벼운 구조로 필요한 상태만 관리할 수 있어 프로젝트의 성능을 높입니다. 전역 상태를 효율적으로
  다루며, 사용법이 간단하고 빠릅니다.
- 사용 이유: 예약 상태, 로그인 상태 등 전역에서 공유할 데이터가 많습니다. Zustand를 사용하면 복잡한 상태 관리 없이 전역
  상태를 손쉽게 다룰 수 있어 개발 생산성이 향상됩니다. 이 스택 조합은 빠르고 유지보수성 높은 코드를 작성할 수 있게
  합니다.

### AWS - 안정적이고 확장 가능한 클라우드 서비스

- 이점: AWS는 클라우드 기반 서비스로, 확장성, 안정성, 보안성을 모두 충족합니다. S3를 사용한 정적 파일 호스팅과
  CloudFront를 통한 CDN 배포로 빠른 웹 성능을 제공합니다.
- 사용 이유: 웨딩 전문가 고용 및 예약 사이트와 같은 프로젝트는 높은 트래픽 처리와 안정적인 데이터 관리가 필수적입니다.
  AWS를 사용하면 전문가 정보와 예약 데이터를 안전하게 저장하고 관리할 수 있습니다. S3와 CloudFront를 통해 빠른 정적 파일
  배포를 제공하며, 글로벌 확장성과 안정성을 보장합니다.

### GitHub Actions - CI/CD 자동화를 위한 워크플로우 플랫폼

이점: GitHub Actions는 코드 변경 사항을 자동으로 테스트, 빌드, 배포까지 연결하는 CI/CD 파이프라인을 제공합니다. YAML
기반의 구성으로 유연하게 설정할 수 있고, GitHub 저장소와 긴밀하게 통합되어 협업이 용이합니다.

사용 이유: GitHub Actions는 웨딩 전문가 고용 및 예약 사이트의 지속적 배포와 품질 관리를 자동화하여 개발 속도와 코드
신뢰성을 높입니다. 특히 jakejarvis/s3-sync-action과 같은 오픈소스 액션을 사용하여 AWS S3에 정적 사이트를 자동 배포할 수
있습니다. 이를 통해 개발자는 기능 개발에 집중할 수 있으며, 안정적인 배포 환경을 유지함으로써 사용자와 전문가 모두에게
안정적이고 빠른 서비스를 제공합니다.

### 브랜치 전략

Git-flow 전략을 기반으로 main, develop 브랜치와 feature 보조 브랜치를 운용했습니다.

main 브랜치는 배포 단계에서만 사용하는 브랜치입니다. featue 브랜치는 기능 단위로, 독립적인 개발 환경을 위하여

## 4. 역할분담

### 손수민 (팀장)

- <b>UI</b>

  - 페이지 :

    - 로그인 페이지 : 서비스를 이용하는 유저가 회원 가입 및 로그인을 진행하는 페이지
    - 메인 페이지 : 해당 서비스를 이용하기 위한 고객이 처음으로 접하게되는 사이트의 메인 페이지

  - 공통 컴포넌트 :
    - Header: 게스트, 유저, 전문가에 따라 헤더 분기 나눔 작업

- <b> 페이지 및 기능 설명 </b>

  - 소셜 로그인 : 구글, 네이버, 카카오로 부터 인가 코드를 제공받아서 백엔드에 post 요청을 보낸 후 그에 대한 응답으로 AT,
    유저 상태, 정보 등을 받아와서 로그인과 동시에 회원 정보를 받아옴. 액세스 토큰 만료 시에 백엔드로 다시 요청을 보내서
    리프레시 토큰이 만료되지 않았다면 AT를 재발급 받고, RT가 만료되었다면 로그아웃 후 로그인 페이지로 리디렉트하여 재
    로그인을 유도함.

    - 구글 소셜 로그인 : 구글 계정을 통한 소셜 로그인 기능 제공
    - 네이버 소셜 로그인 : 네이버 계정을 통한 소셜 로그인 기능 제공
    - 카카오 소셜 로그인: 카카오 계정을 통한 소셜 로그인 기능 제공

  - 로그아웃 : 백엔드 서버에 로그아웃 요청을 보낸 후, 전역으로 관리하는 유저 상태 및 정보를 로그아웃 상태로 변경하고
    세션 스토리지에 저장된 access token 등을 삭제하고 메인 페이지로 리디렉트 처리함.

  - 캐러셀 : 메인 페이지에서 서비스의 정체성을 보여줄 수 있는 사진을 슬라이드 형태의 캐러셀로 보여줌. (React-slick 사용)

  - 전문가 리스트 노출 : 비로그인 상태에서도 다양한 전문가에 대한 정보에 접근할 수 있도록 분야 선택에 따른 전문가
    리스트를 노출함. 탭 클릭 시마다 새로운 전문가 정보를 받아옴.

  - Header 분기처리: 게스트, 유저, 전문가 총 3가지 전역상태를 만들고, 게스트 단에서 보여질 header 메뉴, 유저 상태와
    전문가 상태에 따라 보여질 header 메뉴를 분리하여 보여줌.

### 김수민 (부팀장)

- <b>UI</b>

  - 페이지 :
    - 유저 견적 요청서 페이지 : 일반 고객이 전문가에게 견적서를 작성하고 발송할 수 있는 페이지
    - 마이페이지 > 유저 및 전문가 예약 리스트 페이지 : 유저와 전문가와의 예약 상태를 한눈에 확인 할 수 있는 페이지

  - EstimationList 와 EstimationCard는 사용자가 받은 견적 데이터를 관리하고 표시하는 공통컴포넌트입니다.
    EstimationList는API로 데이터를 불러오고, 카테고리별로 필터링 견적 목록을 탭 UI로 제공합니다.  
    각 견적은EstimationCard를 통해 카드 형식으로 표시 되며, 전문가의 프로필 정보와 요금, 활동 가능 지역
    등의정보를보여줍니다.  
    사용자는 전문가 프로필 보기와 채팅하기 버튼을 클릭하여 상세 정보를 모달로 확인하거나 채팅방으로 이동할 수
    있습니다.  
    EstimationList는 로딩 중에 스피너를 표시하고, useModeChangerStore을 통해 사용자 유형에 따라 접근을 제한하고,
    useModalStore를 통해 상태를 관리하여 사용자 인터페이스를 제공 합니다.

  - Expertlistpage는 전문가가 받은 요청 목록을 관리하고 표시하는 컴포넌트입니다. 이 페이지는 전문가의 요청 데이터를
    API로 불러와 카테고리별로 필터링된 목록을 탭 UI로 제공합니다. 각 요청은 ExpertCard로 표시되며, 전문가의 프로필,
    서비스 카테고리, 활동 가능 지역, 웨딩 일정 등의 정보를 제공합니다. 사용자는 '견적서 보내기' 버튼을 클릭하여 전문가의
    프로필을 모달로 열어 상세 정보를 확인하거나, '받은요청삭제' 버튼을 통해 요청을 삭제할 수 있습니다. 로딩 중에는
    스피너가 표시되고, useModeChangerStore을 통해 전문가만 요청 목록에 접근할 수 있도록 제한하며, useModalStore와
    useCategoryStore를 사용해 상태를 관리하여 사용자 인터페이스를 제공합니다.

    - 채팅 페이지 : 실시간으로 메시지를 주고받을 수 있는 페이지

  - 공통 컴포넌트 :

    - Title: 여러 곳에서 사용하는 공통 타이틀 컴포넌트로, 페이지 및 섹션의 제목을 간결하고 일관되게 표시
    - PageTitle: 페이지의 대표 이름과 함께 뒤로가기 버튼이 포함된 타이틀 컴포넌트로, 페이지 상단에서 사용
    - Button: 다양한 스타일(기본, 강조, 비활성화 등)을 지원하는 공통 버튼 컴포넌트로, 사용자가 작업을 수행하도록 안내
    - IconButton: 아이콘과 함께 사용되는 공통 버튼 컴포넌트로, 주요 액션을 직관적으로 전달
    - Checkbox: 다중 선택이 필요한 상황에서 사용되는 공통 체크박스 컴포넌트로, 상태 변경이 가능
    - Radio: 단일 선택이 필요한 상황에서 사용되는 공통 라디오 버튼 컴포넌트
    - Input: 텍스트 입력을 위한 공통 인풋 컴포넌트로, 다양한 입력 유형(텍스트, 패스워드 등)을 지원
    - Textarea: 멀티라인 텍스트 입력이 필요한 경우 사용되는 공통 텍스트에리어 컴포넌트
    - Select: 드롭다운 형식으로 여러 옵션 중 하나를 선택할 수 있는 공통 셀렉트 컴포넌트
    - Tab: 콘텐츠를 카테고리별로 나누어 표시하는 공통 탭 컴포넌트로, 페이지나 섹션의 전환을 제공
    - Modal: 사용자와의 추가 상호작용을 위해 화면 위에 오버레이 형태로 표시되는 공통 모달 컴포넌트

## 8. 프로젝트 후기

### 손수민 (팀장)

프론트엔드 개발을 시작하고서 처음으로 팀장이라는 직책을 맡게 되어서 부담감도 있었던 건 사실이었지만, 다사다난했던 지난
날을 돌이켜 보면 팀원들이 있었기 때문에 여기까지 올 수 있었던 것 같아 감사한 마음이다. 쏘뉴웨딩 프로젝트를 하면서 기억에
남는 점이 었다면, 소셜 로그인을 처음 시도해보면서 플로우를 잘 이해하지 못해서 많은 시행착오가 있었는데 이번 프로젝트를
마무리하는 시점에서는 소셜 로그인에 대해 알고 이해하게 되었다는 것이다. 또한, 세 팀이 함께 하나의 프로젝트를 완성하기
위해 의기투합하다보니 이런 저런 갈등들이 많이 생겼었다. 이를 해결하기 위해 팀원들 간의 의견을 나누고 그 격차를 줄이는
여러가지 과정을 거치면서 팀장으로써의 태도와 마음가짐, 책임감에 대해 배우고 체득할 수 있었던 뜻깊은 시간이었다.

### 김수민 (부팀장)

팀원들이 서로 의견을 자유롭게 나누는 활동이 가장 기억에 남습니다. 팀원들끼리의 의사소통이 활발히 진행되면 서로 작업물을
확인하면서 팀장이 놓칠 수 있는 부분, 팀원들이 놓칠 수 있는 부분 등의 활동이 쉬워진다는 것을 깨달았습니다. 본인의
작업물만 잘 수행하면 된다는 마음가짐으로는 팀프로젝트 작업물에서 좋은 결과물을 낼 수 없다는 생각을 하게 되었습니다.
그리고 팀을 서포트 하면서 느꼈던 점은 서비스에 필요한 기능 개발 또한 중요하지만 개발 이외의 업무도 중요하다는 것을
깨달았습니다. 서비스 제작에 필요한 와이어프레임, 화면정의서, 요구사항 정의서, 플로우 차트가 제대로 준비되지 않고, 숙지
또한 되어있지 않으면 개발도중에 기획회의를 진행할수 있다는 점, 불필요한 회의가 잦아질수 있다는 점을 알게되었습니다.
개발자에게 필요한 제 1덕목은 개발이겠지만 그 이외의 역량도 중요하다는 것을 깨닫고 개발을 진행하는 도중에도 개인 작업물
기록을 포함하여 개발관련 문서에 문제는 없는지 수시로 체크하도록 습관을 길러야 겠다고 생각했습니다.

### 이상민 (팀원)

개발을 하면서 처음에는 쉽게 쉽게 할 수 있는 부분들이 많을 거라고 생각을 했지만 서로의 연결부분이 까다롭고 들어오는
데이터가 많기에 잘 생각하고 연결하지 않으면 연결이 되지 않는다는 것을 알았고 이번에 맡은 파트는 벡엔드와의 조율이 많이
들어가는 부분이 많아 트러블도 조금 생기고 의결 불합도 많이 생겼지만 의견에 대해 어떠한 부분의 조율이 필요한지 알아갔고
많은 노력을 통해 만들어 갔다고 생각합니다.

### 김대식 (팀원)

웹소켓에 대한 지식이 하나도 없는 상태에서 도전해 보자는 생각으로 무작정 채팅/알람 파트를 맡아 진행을 했었는데 처음에는
괜히 도전했나라는 생각도 들었지만 팀원들과 조교님이랑 새벽까지 대화하며 기능 구현을 완성하고 나니 도전하길 잘 한 것
같다.

### 박미선 (팀원)

전문가 프로필 등록 및 수정 페이지를 초기에 맡았을 땐 간단한 페이지라 생각하고 얼른 끝내서 다른 페이지를 해야겠다는
생각이었지만 하다보니 이것저것 예외처리 해야 할 부분이 생기면서 작업하는 동안 재미있었다.
